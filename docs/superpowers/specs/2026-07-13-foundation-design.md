# Foundation Design Spec — Tafsir Nabulsi Project

**Date:** 2026-07-13  
**Scope:** Phase A — Python data pipeline + Supabase schema  
**Sequencing:** Foundation → Web app → Android app

---

## Overview

Two companion apps (React web + Kotlin Android) share:
- **Content**: per-surah JSON files generated by a Python pipeline
- **User data**: shared Supabase instance (auth, bookmarks, reading_progress)

This spec covers the shared foundation both apps depend on.

---

## Part 1 — Python Data Pipeline

### Location
`pipeline/` in the `tafseer-nabulsi` repo.

### Tooling
- `uv` for venv + dependency management
- Python 3.12+
- `pyproject.toml` with deps: `beautifulsoup4`, `requests`, `lxml`, `pytest`

### Output schema (per-surah JSON: `1.json`…`114.json`)
```json
{
  "surah_id": 1,
  "name": "الفاتحة",
  "ayahs": [
    {
      "number": 1,
      "text": "بِسْمِ اللَّهِ الرَّحْمَٰنِ الرَّحِيمِ",
      "tafsir_short": "الدرس: تفسير الآية 1 — المقدمة",
      "tafsir_long": "نص التفسير الكامل...",
      "media": {
        "audio_url": "",
        "video_url": ""
      }
    }
  ]
}
```

### Data sources
| Source | Format | Attribution |
|---|---|---|
| Tanzil.net | Uthmani XML (v1.1, Feb 2021) | Display source + link to tanzil.net |
| nabulsi.com | Scraped lesson HTML | Respectful scraping, 1 req/sec |
| media.csv | User-maintained CSV | Optional audio/video URLs |

### Module structure
```
pipeline/
├── pyproject.toml
├── media.csv.example
├── src/
│   ├── main.py                 CLI entry: --surah N | --all | --resume | --report
│   ├── config.py               Constants: URLs, paths, rate limits
│   ├── quran/
│   │   ├── fetcher.py          Download + disk-cache Tanzil XML
│   │   └── parser.py           XML → list[Surah]
│   ├── tafsir/
│   │   ├── surah_index.py      114 surah names → category URL slugs
│   │   ├── scraper.py          Polite fetch + disk-cache of lesson pages
│   │   ├── lesson_parser.py    Arabic title regex → ayah range
│   │   └── content_extractor.py Extract body text from HTML
│   ├── media/mapper.py         Load media.csv, map_media_links()
│   ├── merge/builder.py        Merge quran+tafsir+media → JSON schema
│   └── utils/
│       ├── cache.py            Disk-based HTTP response cache
│       ├── rate_limit.py       1 req/sec delay
│       └── logging_setup.py    Structured logging
├── tests/
│   ├── test_lesson_parser.py
│   ├── test_builder.py
│   └── fixtures/
└── output/                     Generated JSON + _index.json + _report.json
```

### Key design: Arabic title → ayah range mapping
nabulsi.com lesson titles embed ayah references:
- `الآية 24` → ayah 24
- `الآيتان 25-26` → ayahs 25, 26
- `الآيات 1-5` → ayahs 1, 2, 3, 4, 5

`lesson_parser.py` uses Arabic regex to extract these ranges.

### Guarantees
- 1 req/sec rate limit + exponential backoff
- Disk cache for all HTTP responses (resumable)
- Per-surah checkpoint (failures don't restart from scratch)
- Gap report: lists ayahs with no tafsir coverage
- Tanzil attribution carried into `_index.json`

## Part 2 — Supabase Schema

### Location
`supabase/` in the `tafseer-nabulsi` repo.

### Tables

```sql
profiles
  id          uuid PK (refs auth.users.id ON DELETE CASCADE)
  email       text
  display_name text

bookmarks
  id           uuid PK DEFAULT gen_random_uuid()
  user_id      uuid refs profiles.id ON DELETE CASCADE
  surah_id     int (1-114)
  ayah_number  int
  note         text NULL
  created_at   timestamptz DEFAULT now()
  UNIQUE(user_id, surah_id, ayah_number)

reading_progress
  user_id           uuid refs profiles.id ON DELETE CASCADE
  surah_id          int (1-114)
  last_ayah_number  int
  updated_at        timestamptz DEFAULT now()
  PK(user_id, surah_id)
```

### RLS policies
Every table: `USING (auth.uid() = user_id)` for all operations.

### Triggers
- `handle_new_user()`: on `auth.users INSERT` → creates `profiles` row
- `set_updated_at()`: auto-bumps `updated_at` on reading_progress update

### Indexes
- `bookmarks(user_id)`
- `reading_progress(user_id, surah_id)`
